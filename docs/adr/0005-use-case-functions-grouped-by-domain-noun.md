# Application use-case functions are grouped by Domain noun, not by command

Each Domain noun gets its own Application module: `bookings.py` (`add`), `blocked.py` (`add`, `remove`), `watches.py` (`add`, `list_all`, `remove`, `run_all`), `catalog.py` (`find_park`, `resolve_park`, `list_parks_filtered`), `search.py` (`run`, `build_profile`), etc. Use-case logic is **functions, not UseCase classes** — Typer commands and the daemon Poller call these functions directly, so a class with one method and no state would be ceremony. CLI commands stay thin: parse arguments, call the use-case function, format the result. Inverse rule: trivial DB-only commands (`watch list`, `bookings rm`, `blocked list`) skip the use-case layer and call `store.*` directly — wrapping `store.list_watches()` in a use-case adds nothing.
